Solving Programming Problems

"Problem-solving [is] solving problems is the core of computer science. Programmers must first understand how a human solves a problem, then understand how to translate this "algorithm" into something a computer can do, and finally how to "write" the specific syntax (required by a computer) to get the job done."

Problem? Problem... Not a problem! Maybe...
Sometimes a problem is not a problem at all (Thank you, Captain Obvious!) When I'm reading the first few lines in a problem, I often think: "I have NO IDEA what to do!". That's my panic point. But! I know I need to stay calm and read the task several times, until fully understand it. Well, it doesn't work in my case sometimes. My example is the deBee function. I thought I understood the problem and I wrote the function without using filter() and toLowerCase() methods. Then, I wrote the second deBee function using toLowerCase() method, but without filter(). And only the third deBee function I've wrote was with filter() and toLowerCase() methods. All three functions are doing the same work, but the last one is shorter and more elegant, just 3 lines of code. I'm happy :)
Me and Using the Problem Solving Techniques: Some Reflections
Pseudocode
I can't say I often use Pseudocode technique. Maybe it would be better to use it more frequently, because Pseudocode is a detailed description of what a program must do, this can help me write every line of code needed in order to solve the problem.
Trying something in Console
I like trying code in Console. I take very simple examples and find the output. Then I take more complex and bigger inputs to see what will be the output. Console is a good tool for experiments with code. I'm using debugger to see what code is doing. Debugging helps me catch any syntax errors or gaps in logic.
Rubber ducky method
The best way to know if you understand the problem is by being able to explain it to someone else. So... Sometimes I ask my husband to be my Rubber Ducky. He is a brilliant one!
Googling
I google not to find a solution, but to find ideas and check if methods I chose work as I'm expecting.
Asking your peers for help
I think there is absolutely nothing wrong in asking for help to solve a programming problem. Every programmer needs help time to time. I haven't ask my peers for help yet, but I will if I need it.
Asking coaches for help
I haven't asked yet, but I feel confident to do it.
Improving your process with reflection
I'm doing reflection after each task. Especially if task was difficult for me. Reflection helps me analyse what I've done and understand what can be improved.
